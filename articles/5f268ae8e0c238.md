# 0. Summary

- 環境
    - kvmホストマシン から KVM ゲストマシン に、VSCODEのSSHで接続している
    - KVMホストマシン ubuntu 24.04 desktop
    - KVMゲストマシン: ubunt 24.04 server
- 事象
    - VS Code から Codex につなげようとすると時間がかかり過ぎで表示されない

    - ローカルのvscodeの設定がなんらかおかしくなっていた

# 1. 事象
- VS Code 上で、 OUTPUT(terminal を開くとその左にあるタブから開ける)を開き、codexの出力を確認すると以下の出力

```
> [error] [CodexMcpConnection] cli: message="codex_app_server: Codex could not find bubblewrap on PATH. Install bubblewrap with your OS package manager. See the sandbox prerequisites: https://developers.openai.com/codex/concepts/sandboxing#prerequisites. Codex will use the bundled bubblewrap in the meantime."
```

そこで、bubblewrapのインストール
```
sudo apt update
sudo apt install -y bubblewrap
```

これまでBUBBLEWRAPとやらがなくても動いていたのに、なぜ？

可能性が高いのは、Codex CLI / VS Code Codex拡張が更新され、Linux sandboxの既定実装・起動時チェックが bubblewrap 前提に寄ったため
必要になった、というよりも警告が見えるようになったというのが正しい

確認結果から、Ubuntu 24.04のAppArmorによる unprivileged user namespace 制限によるもの
開発用マシンなので、APPARMORの制限を外す

```
echo 'kernel.apparmor_restrict_unprivileged_userns=0' | sudo tee /etc/sysctl.d/99-codex-apparmor-userns.conf
sudo sysctl --system
```


これが、emote-SSHで報告されている事象とほぼ完全に一致。同じ事例では、app-serverは生きているのにCodex WebViewだけがロード状態から進まないということになっている
[こちら](https://github.com/openai/codex/issues/26951)

最後に、KVMホストマシン（sshの接続元となるマシン）のvscodeを再インストールすることとした


```bash
curl https://packages.microsoft.com/keys/microsoft.asc | gpg --dearmor > microsoft.gpg
ls -l /etc/apt/trusted.gpg.d/
sudo install -o root -g root -m 644 microsoft.gpg /etc/apt/trusted.gpg.d/
ls -l /etc/apt/trusted.gpg.d/
ls /etc/apt/sources.list.d/
sudo sh -c 'echo "deb [arch=amd64] https://packages.microsoft.com/repos/vscode stable main" > /etc/apt/sources.list.d/vscode.list'
ls -l /etc/apt/sources.list.d/

cat /etc/apt/sources.list.d/vscode.sources
sudo apt update
sudo apt list -a code
sudo apt install code
which code
```
